# Feature Specification: Ambient 模式说话人识别

**Feature Branch**: `017-ambient-speaker-id`  
**Created**: 2026-04-15  
**Status**: Draft  
**Input**: 基于 `docs/ambient-speaker-identification-proposal.md` 可行性方案

## 背景与动机

### 问题定义

当前 Ambient 模式下，所有语音均被视为来自 WebSocket 连接用户，无法区分"谁在说话"。这导致：

1. **多人场景失效**：家庭中多人同时存在时，所有语音都归属于同一用户，上下文记忆、个性化回复完全错乱
2. **TTS 回声浪费**：系统播放的 TTS 回复被麦克风拾取后再次送入 ASR，触发无意义的 Agent 推理，浪费 LLM token
3. **体验割裂**：前端无法显示"这句话是谁说的"，与 015-family-multiuser 的多用户设计脱节

### 核心洞察

之前 diarization（盲聚类）方案失败的根因是**选错了问题类型**。家庭场景中成员是已知的，应使用 Speaker Identification（预注册识别）而非 Speaker Diarization（无监督聚类）。前者准确率 95%+，后者在远场+噪声+短语句叠加下仅 60-80%。

### 已有基础设施

系统已具备完整的声纹基础设施（声纹数据模型、声纹注册/识别服务、per-speaker 聚合器、前端 speaker 状态字段），但从未在 ambient 路径中启用。本特性的核心是**激活已有能力**，而非构建新系统。

## Clarifications

### Session 2026-04-15

- Q: Ambient 模式下"同一会话"的定义（临时标签生命周期）？ → A: 永不重置——同一未知说话人始终保持相同标签，直到其注册声纹后自动替换为真实身份。标签需持久化存储，跨 ambient 会话和 WebSocket 连接保持一致。
- Q: FR-013 回溯匹配历史消息的时间范围？ → A: 全部历史，不限时间。新用户注册声纹后，系统匹配所有未识别的历史消息，而非仅限近期。

## User Scenarios & Testing

### User Story 1 - 已注册家庭成员说话被自动识别 (Priority: P1)

安琳和团子爸爸都已注册声纹。Ambient 模式开启后，安琳对着 reSpeaker 说"帮我查一下明天天气"，系统自动识别出是安琳在说话，使用安琳的上下文记忆和个性化设置进行回复。随后团子爸爸说"打开客厅灯"，系统识别出是团子爸爸，切换到他的用户上下文处理请求。

**Why this priority**: 这是整个特性的核心价值——让 Ambient 模式从"单用户"进化为"多用户感知"，直接关联 015 家庭多用户体系。

**Independent Test**: 注册 2 个家庭成员声纹，在 ambient 模式下分别说话，验证系统正确识别说话人并使用对应用户的上下文进行回复。

**Acceptance Scenarios**:

1. **Given** 安琳已注册声纹且 ambient 模式开启，**When** 安琳说"帮我查一下明天天气"，**Then** 系统识别为安琳，使用安琳的记忆和偏好生成回复
2. **Given** 团子爸爸已注册声纹且 ambient 模式开启，**When** 团子爸爸说"打开客厅灯"，**Then** 系统识别为团子爸爸，使用团子爸爸的权限执行智能家居操作
3. **Given** 安琳和团子爸爸交替说话，**When** 连续 10 轮交替对话，**Then** 每轮都能正确识别说话人，识别准确率 >= 90%

---

### User Story 2 - TTS 回声被自动过滤 (Priority: P1)

系统通过小爱音箱播放 TTS 回复时，reSpeaker 麦克风会拾取到这些声音。系统需要自动识别并丢弃这些"回声"，避免触发无意义的 Agent 推理。

**Why this priority**: 与 P1 并列最高优先级。如果不解决回声问题，每次 TTS 播放都会触发新一轮推理，造成无限循环，直接导致 ambient 模式不可用且大量浪费 LLM token。

**Independent Test**: 在 ambient 模式下触发一次 TTS 播放，验证 TTS 内容被 ASR 拾取后不会触发新的 Agent 推理。

**Acceptance Scenarios**:

1. **Given** 系统正在播放 TTS 回复，**When** 麦克风拾取到 TTS 声音并被 ASR 转录，**Then** 系统识别为回声并丢弃，不触发 Agent 推理
2. **Given** 系统刚播放完 TTS 回复（5 秒内），**When** ASR 转录出与 TTS 内容高度相似的文本，**Then** 系统识别为回声尾音并丢弃
3. **Given** 系统正在播放 TTS 回复，**When** 用户同时说了一句与 TTS 内容完全不同的话，**Then** 用户的话不被误过滤，正常触发 Agent 处理

---

### User Story 3 - 前端显示说话人标识 (Priority: P2)

在 ambient 模式的对话界面中，每条语音消息旁显示说话人信息：已注册用户显示头像和用户名，未注册用户显示数字标签（如"用户01"）。

**Why this priority**: 提升用户体验，让家庭成员清楚看到对话归属，但不影响核心识别功能的运行。

**Independent Test**: 在 ambient 模式下进行对话，检查前端消息气泡是否正确显示说话人头像/名称。

**Acceptance Scenarios**:

1. **Given** 安琳（已注册）在 ambient 模式下说话，**When** 消息显示在前端，**Then** 消息气泡旁显示安琳的头像和用户名
2. **Given** 一位未注册访客在 ambient 模式下说话，**When** 消息显示在前端，**Then** 消息气泡旁显示数字标签圆圈（如"01"）和"用户01"
3. **Given** 两位不同的未注册访客先后说话，**When** 消息显示在前端，**Then** 两人分别显示不同的数字标签（"用户01"和"用户02"）

---

### User Story 4 - 未注册访客语音的临时标签管理 (Priority: P3)

当未注册声纹的访客说话时，系统为其分配临时标签（unknown_01, unknown_02...），同一访客在同一会话中始终使用相同标签。若该访客之后注册了声纹，系统可回溯匹配并更新历史消息的说话人信息。

**Why this priority**: 完善边缘场景体验，但不影响已注册家庭成员的核心使用流程。

**Independent Test**: 让未注册用户在 ambient 模式下说几句话，验证标签一致性；然后为该用户注册声纹，验证历史消息是否更新。

**Acceptance Scenarios**:

1. **Given** 一位未注册访客在 ambient 模式下连续说了 3 句话，**When** 系统处理这些语音，**Then** 3 句话都归属于同一临时标签"unknown_01"
2. **Given** 两位不同的未注册访客交替说话，**When** 系统处理这些语音，**Then** 两人分别获得不同的临时标签"unknown_01"和"unknown_02"
3. **Given** 之前被标记为"unknown_02"的访客注册了声纹，**When** 注册完成，**Then** 该访客所有历史消息（不限时间范围）自动更新为其真实用户名

---

### Edge Cases

- **极短语音（< 0.5 秒）**：如"嗯"、"啊"等感叹词，音频太短无法可靠识别说话人时，系统应跳过识别，使用默认用户或标记为"未识别"
- **多人同时说话**：两人声音重叠时，系统取主要声源（信号最强）进行识别，不尝试分离
- **识别服务不可用**：声纹识别服务暂时不可用时，系统应降级到现有行为（所有语音归属 WebSocket 连接用户），不中断 ambient 基础功能
- **TTS 回声与真实语音混合**：系统播放 TTS 的同时用户说了不同的话，需要尽量保留用户的真实语音
- **声纹置信度边界**：识别结果置信度处于阈值附近时（如 0.58-0.62），需要合理的处理策略（降级为未识别而非错误归属）

## Requirements

### Functional Requirements

- **FR-001**: 系统必须在 ambient 模式下的每段语音转录完成后，自动进行说话人识别
- **FR-002**: 系统必须将识别到的说话人映射到对应的 LinChat 用户，并使用该用户的上下文记忆和偏好进行 Agent 推理
- **FR-003**: 系统必须在 TTS 播放期间及播放结束后 5 秒窗口内，自动检测并丢弃 ASR 拾取的回声内容
- **FR-004**: TTS 回声检测必须采用双重验证策略（播放状态 + 文本相似度比对），降低误判风险
- **FR-005**: 系统必须为未注册声纹的说话人分配持久化临时标签，同一说话人跨多次 ambient 会话和 WebSocket 连接始终保持相同标签，直到该说话人注册声纹后自动替换为真实身份
- **FR-006**: 系统必须通过 WebSocket 事件将说话人识别结果实时推送给前端
- **FR-007**: 前端必须根据说话人身份显示对应的头像和用户名（已注册用户）或数字标签（未注册用户）
- **FR-008**: 说话人识别过程必须异步执行，不阻塞 ASR 流式转录的主路径
- **FR-009**: 系统必须提供功能开关，关闭后恢复 ambient 模式的原有行为（所有语音归属连接用户）
- **FR-010**: 音频长度低于 0.5 秒时，系统应跳过说话人识别
- **FR-011**: 识别服务不可用时，系统必须降级到原有行为，不中断 ambient 基础功能
- **FR-012**: 声纹识别的置信度阈值必须可配置
- **FR-013**: 新用户注册声纹后，系统必须回溯匹配所有未识别的历史消息（不限时间范围）并更新说话人信息

### Key Entities

- **说话人档案 (SpeakerProfile)**: 代表一个已注册声纹的家庭成员，关联 LinChat 用户账户，包含声纹注册 ID 和用户名
- **说话人识别结果**: 每段语音的识别输出，包含是否识别成功、匹配到的用户、置信度分数
- **临时说话人标签**: 未注册说话人的持久化标识（非会话级），基于声纹特征区分不同未知说话人，跨 ambient 会话保持一致，注册声纹后自动替换
- **TTS 播放状态**: 记录当前是否有 TTS 正在播放以及最近播放的文本内容，用于回声检测

## Scope & Constraints

### 范围内

- Ambient 模式下的说话人自动识别（仅 ambient 路径，不影响 voice_chat 模式）
- TTS 回声检测与过滤
- 前端说话人标识显示
- 未注册说话人临时标签管理
- 功能开关与降级机制

### 范围外（不在本期）

- 硬件固件改动（ESP32-S3 / reSpeaker XVF3800 保持现状）
- 新增 ML 模型或框架（复用已有声纹识别能力）
- 端侧 VAD 过滤优化
- 流式 diarization
- 端侧唤醒词检测

### 假设

- 家庭成员已通过现有声纹注册流程完成注册
- 声纹识别服务（Gateway API）稳定可用，单次识别延迟 < 200ms
- reSpeaker XVF3800 的波束成形和降噪已提供足够质量的音频
- 家庭场景下同时在场的说话人数量通常 <= 5 人

### 依赖

- 015-family-multiuser（家庭多用户体系，已完成）
- 016-respeaker-wifi-ambient（reSpeaker WiFi 环境监听，开发中）
- Gateway 声纹识别 API（`/v1/voice/speakers/identify`，已生产就绪）

## Success Criteria

### Measurable Outcomes

- **SC-001**: 已注册家庭成员在 ambient 模式下的说话人识别准确率 >= 90%（10 分钟连续对话测试）
- **SC-002**: TTS 回声过滤率 >= 95%（100 次 TTS 播放中，被 ASR 拾取的回声被成功过滤的比例）
- **SC-003**: TTS 回声过滤误杀率 <= 2%（100 次真实用户发言中，被误判为回声的比例）
- **SC-004**: 说话人识别不增加用户感知延迟（从语音结束到系统开始回复的时间与未开启识别时差异 < 500ms）
- **SC-005**: 功能开关关闭后，ambient 模式行为与开启前完全一致（零回归）
- **SC-006**: 未注册说话人的临时标签在同一会话中一致性达 100%（同一人始终获得相同标签）
- **SC-007**: 前端正确显示说话人标识（已注册用户头像+用户名，未注册用户数字标签）
