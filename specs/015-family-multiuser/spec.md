# Feature Specification: 家庭多用户系统

**Feature Branch**: `015-family-multiuser`
**Created**: 2026-03-11
**Status**: Draft
**Input**: 家庭多用户系统 — SysUser 模型扩展、成员管理面板、访客机制、声纹注册 UI

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 成员查看和管理家庭 (Priority: P1)

成员登录 LinChat 后，底部输入框左侧显示当前用户头像和切换按钮。点击后弹出模态框，框内展示可切换的用户列表（有效用户正常展示，过期访客灰色展示在列表底部）和添加用户入口。成员可通过分步引导创建新用户（步骤1：选择类型 成员/访客 + 用户名密码；步骤2：注册声纹）。所有用户一旦创建不可删除。访客登录后无切换按钮和管理入口。

**Why this priority**: 成员管理是整个家庭多用户体系的基础。没有成员管理，其他所有功能（访客、声纹、语音检查）都无法运作。

**Independent Test**: 成员登录后点击切换按钮打开模态框，执行创建操作，验证列表实时更新；访客登录后验证切换按钮不可见。

**Acceptance Scenarios**:

1. **Given** 成员已登录, **When** 点击底部输入框左侧的切换按钮, **Then** 弹出模态框，显示有效用户列表（每人显示头像首字母、用户名、类型标签）和"添加用户"入口；过期 guest 在列表底部以灰色"已过期"样式展示（仅供查看状态，不可点击切换）
2. **Given** 成员在模态框中, **When** 点击"添加用户"并完成步骤1（选择 成员/访客 类型 + 填写用户名密码）, **Then** 进入步骤2 声纹注册界面（声纹注册流程见 US4）
3. **Given** 访客已登录, **When** 查看界面, **Then** 底部输入框左侧无切换按钮，无任何管理入口

---

### User Story 2 - 成员快速切换到其他用户视角 (Priority: P1)

成员在模态框的用户列表中直接点击目标用户，即可切换聊天对话的发言人到该用户，查看其聊天记录。切换不改变登录身份，仅改变聊天消息读写和 AI 回复（含记忆召回）的目标用户；其他 API（记忆管理、语音设置、媒体上传等）始终使用登录用户自身数据。切换后模态框关闭，页面刷新展示目标用户的独立聊天数据。

**Why this priority**: 家庭场景下成员（家长）需要快速查看访客或其他成员的使用情况，快速切换是核心交互体验。

**Independent Test**: 成员在模态框中点击目标用户后，页面刷新为目标用户的聊天界面，消息历史完全属于目标用户，成员代发的消息 AI 使用目标用户的记忆上下文回复；登录身份不变，管理权限保持；其他功能（记忆管理、语音设置等）仍使用登录用户自身数据。

**Acceptance Scenarios**:

1. **Given** 成员已打开用户切换模态框, **When** 点击用户 A, **Then** 系统将聊天对话的发言人切换为用户 A（不颁发新令牌），模态框关闭，页面刷新为用户 A 的聊天界面；成员代发的消息 AI 使用用户 A 的记忆上下文回复
2. **Given** 已切换到用户 A 视角, **When** 查看聊天历史, **Then** 仅显示用户 A 的消息，不显示其他人的消息
3. **Given** 已切换到用户 A 视角, **When** 成员再次点击切换按钮, **Then** 模态框正常弹出，管理功能仍可用（权限基于登录身份，不受切换影响）
4. **Given** 成员尝试切换到过期访客, **When** 查看列表, **Then** 过期访客灰色禁用态，点击无响应

---

### User Story 3 - 访客临时访问 (Priority: P2)

成员可以创建一个访客账号供短期来访的人使用。访客账号有有效期，过期后自动失效。访客在有效期内仅可使用聊天功能，无法查看成员管理面板、创建/删除任何用户、切换聊天对话发言人等管理操作。

**Why this priority**: 访客机制扩展了系统的灵活性，但不是日常使用的核心流程。

**Independent Test**: 创建访客账号 → 访客登录使用聊天 → 验证无管理入口 → 过期后验证无法登录。

**Acceptance Scenarios**:

1. **Given** 成员在添加成员弹窗中, **When** 选择类型为"访客"并提交, **Then** 访客账号创建成功，默认有效期 7 天，列表中显示"访客"标签
2. **Given** 访客账号已创建, **When** 访客使用该账号登录, **Then** 可正常使用聊天功能，但不可见成员管理面板，不可创建其他用户
3. **Given** 访客已登录, **When** 尝试调用管理类接口（创建成员/切换用户等）, **Then** 系统拒绝请求并返回权限不足错误
4. **Given** 访客账号已过期, **When** 访客尝试登录, **Then** 登录失败，提示账号已过期
5. **Given** 系统定时检查, **When** 发现过期访客, **Then** 自动标记为不可用

---

### User Story 4 - 声纹注册 (Priority: P2)

声纹注册作为用户创建流程的必要步骤2（不可跳过）。注册过程为：点击录音按钮 → 朗读一段话（最低 10 秒，最长 30 秒）→ 录音完成后，向导将全部信息（用户名、密码、类型、音频）一次性提交后端。后端原子完成声纹注册（Gateway）和用户入库，任一失败则整体失败可重试，不可跳过。由于初始管理员自带声纹、且创建为原子操作（声纹+用户同时入库），系统中不存在无声纹的用户。

**Why this priority**: 声纹注册是语音模式的前置条件，但文字聊天功能不依赖声纹。

**Independent Test**: 用户点击录音 → 完成 30 秒内录音 → 提交创建 → 创建成功（声纹+用户原子入库）→ 用户出现在成员列表。

**Acceptance Scenarios**:

1. **Given** 用户尚未注册声纹, **When** 打开声纹注册界面并点击录音, **Then** 开始录音，显示倒计时进度条（最低 10 秒，最长 30 秒），不足 10 秒时提交按钮禁用
2. **Given** 录音进行中, **When** 用户点击停止或录音达到 30 秒, **Then** 录音停止，向导显示"提交创建"按钮
3. **Given** 用户点击提交, **When** 后端原子创建成功（声纹注册+用户入库）, **Then** 界面显示"创建成功"，成员列表实时更新显示新用户
4. **Given** 用户点击提交, **When** 后端创建失败（如声纹音质不够或用户名重复）, **Then** 显示具体错误提示，用户可重新录制或修改信息后重试

---

### User Story 5 - Web 语音模式统一为 ambient 模式 (Priority: P3)

浏览器端发起语音连接时，统一使用 ambient 模式（014-jarvis 已实现的环境监听模式），不再区分 voice_chat 和 ambient 两种模式。

**Why this priority**: 模式统一是架构优化，用户无感知但简化了代码维护。依赖其他核心功能先就绪。

**Independent Test**: 浏览器发起语音连接 → 后端收到的 mode 参数为 ambient → 语音交互正常。

**Acceptance Scenarios**:

1. **Given** 用户在浏览器端点击语音模式, **When** 连接建立, **Then** 以 ambient 模式连接后端，语音交互正常
2. **Given** 成员已切换到其他用户视角, **When** 点击语音模式按钮, **Then** 自动恢复为自身视角（调用 clearTarget 并重新加载自身聊天历史，语音模式使用登录用户声纹），显示提示"语音模式需使用本人声纹，已切换回自身视角"，然后正常建立 ambient 连接

### Edge Cases

- 成员切换聊天到访客后，访客恰好过期？ — 系统禁止切换到过期访客。若切换后访客过期（极端时序），中间件返回错误响应（错误码 TARGET_USER_INVALID，提示"该用户已过期，无法继续查看"），前端收到错误后自动清除 targetUserId 并恢复自身视角
- 声纹录音中浏览器失去焦点或网络断开？ — 录音中断时提示用户重新录制
- 创建用户向导中途退出（浏览器崩溃/用户关闭页面）？ — 数据库无任何残留（前端分步收集数据、后端一次性原子创建），用户可随时重新启动创建向导，用户名不被占用
- 成员切换视角时正在进行 AI 回复（SSE 流）？ — 前端切换时中断当前 SSE 流（复用现有停止按钮逻辑），然后清空消息并加载目标用户聊天历史
- 成员切换聊天到自己？ — 忽略操作（无任何提示，不触发页面刷新）
- 访客尝试调用管理 API？ — 后端接口层校验 member_type，访客请求一律返回 403
- 创建用户时用户名重复？ — 数据库 UNIQUE 约束保证用户名唯一，提示"用户名已存在"，需选择不同用户名
- 声纹数据是否过期？ — 声纹永久保存，不设过期机制。声纹是家庭成员的永久身份标识（未来 ESP 硬件麦克风场景依赖声纹识别说话人），直到全量清库才会删除
- 用户能否删除？ — **不能**。所有用户一旦创建不可删除，该类型问题不予考虑。访客过期后仅禁用（status=0），不从系统中移除
- 是否存在无声纹的用户？ — **不存在**。初始管理员自带声纹；新用户创建为原子操作（用户入库与声纹注册在同一请求中完成，任一失败则全部不入库），物理上不可能存在无声纹用户
- 开发完成后数据如何处理？ — 全量清库：清空 PostgreSQL 所有业务表、MinIO 存储桶、向量库、Langfuse 监控数据，然后初始化带声纹的管理员测试账户。无需存量数据迁移

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 使用用户名（username，3-50 字符，仅字母数字下划线，大小写敏感完全匹配，唯一——即 `Anlin` 与 `anlin` 为不同用户，由 PostgreSQL 数据库级 UNIQUE 约束保证）作为唯一显示名称，头像首字母取自用户名首字符大写（如 dantsinghua → D，123user → 1，_test → _）；非字母字符不做大写转换，直接使用原字符；不引入额外的 display_name 字段
- **FR-002**: 系统 MUST 区分成员（member）和访客（guest）两种业务类型
- **FR-003**: 系统 MUST 为访客账号设置固定 7 天有效期（从创建时间起算），过期后自动失效（登录时返回 400 ACCOUNT_EXPIRED 拒绝——密码验证已通过属业务拒绝而非认证失败；已登录访客过期后，中间件在下次请求时返回 401 终止会话——此时属令牌失效），但永久保留其聊天和记忆数据
- **FR-004**: 成员 MUST 能够创建家庭成员和访客（不支持编辑和删除，用户一旦创建不可删除）；创建时密码经 SM4 解密后 MUST 为 6-50 个 ASCII 可打印字符（0x20-0x7E，含字母、数字、空格、特殊符号，不允许中文及其他非 ASCII 字符）（访客权限限制见 FR-010）；用户创建为原子操作：前端分步收集全部信息（步骤1 类型+凭据、步骤2 声纹录音），完成后一次性提交 `POST /api/v1/members/`（multipart/form-data），后端在单个请求中完成声纹注册（Gateway API）和用户入库（status=1），任一环节失败则整体失败、数据库无残留
- **FR-005**: 成员 MUST 能够切换当前聊天对话的发言人到任意有效（status=1 且未过期）的用户（不改变登录身份，不颁发新令牌）；切换仅影响聊天消息读写（含消息附件下载）和 AI 回复（含记忆召回），其他 API（记忆管理、语音设置、媒体上传等）始终使用登录用户；切换后目标 user_id 持久化到 localStorage + Zustand store（刷新保持、新标签页打开时通过 localStorage 初始化读取继承上次切换状态，非跨标签页实时同步）；访客 MUST NOT 能够执行切换操作
- **FR-006**: 用户创建流程 MUST 包含声纹注册步骤（不可跳过），通过浏览器录音（最低 10 秒，最长 30 秒）完成；前端向导步骤2录制音频后，将全部信息（用户名、密码、类型、音频）一次性提交后端原子创建；录音不足 10 秒时禁用提交；创建失败（含声纹注册失败）可重试
- **FR-007**: Web 端语音模式 MUST 统一使用 ambient 模式连接后端
- **FR-008**: 系统 MUST 定期检查并自动停用过期的访客账号（FR-003 过期策略的实现机制：Celery beat 每小时扫描）
- **FR-009**: 各用户的聊天数据、记忆数据 MUST 完全按 user_id 隔离；切换后聊天界面展示目标用户的独立聊天数据，AI 回复使用目标用户的记忆上下文；其他管理页面始终展示登录用户自身数据
- **FR-010**: 访客 MUST NOT 能够访问成员管理面板、创建用户、切换聊天对话发言人；后端 MUST 对所有管理类接口校验调用者的 member_type（基于 `request.user_id` 即登录用户），访客请求一律拒绝
- **FR-011**: 系统 MUST 禁止成员切换到过期访客（前端列表中过期访客灰色展示、不可点击）
- **FR-012**: 用户一旦创建 MUST NOT 可被删除（系统不提供删除功能，该类型问题不予考虑）
- **FR-013**: 系统中不存在无声纹的用户（初始管理员自带声纹 + 创建为原子操作确保声纹与用户同时入库）。此为系统不变量，不作为功能开发；若因异常违反此不变量，由前后端统一错误处理机制（宪法 1.4 显式失败原则）返回错误，无需专门的声纹前置检查功能
- **FR-014**: 创建用户向导中途退出（步骤1或步骤2未完成）时，数据库 MUST NOT 有任何残留状态——向导仅在前端收集数据，后端原子创建确保用户要么完整存在（status=1 + 声纹已注册）要么完全不存在；过期访客被 Celery 标记为 status=0 后 MUST NOT 出现在成员列表或可切换用户列表中（status=0 用户不在任何 API 返回结果中）

### Key Entities

- **家庭成员（SysUser 扩展）**: 现有 SysUser 即家庭成员，新增业务类型（member_type: member/guest）、过期时间（guest_expires_at）。用户名即显示名，头像首字母取自用户名首字符大写（不存储）。ID 为 SysUser 表自增主键。用户一旦创建不可删除
- **声纹档案（SpeakerProfile）**: 已有实体，与 SysUser 一对一关联。所有用户均有声纹（创建流程强制注册）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 成员可在 1 分钟内完成新用户创建 Step 1（选择类型 + 填写用户名、密码、提交），声纹注册耗时由 SC-003 单独衡量
- **SC-002**: 成员快速切换用户身份后，页面在 3 秒内完成刷新并展示目标用户的数据
- **SC-003**: 声纹注册全流程（录音 + 上传 + 确认）可在 1 分钟内完成
- **SC-004**: 访客账号过期后登录时立即拒绝；定时任务每小时扫描并标记过期账号（双重保障），无需人工干预
- **SC-005**: 数据隔离准确率 100% — 切换后仅展示目标用户的独立聊天和记忆数据，不出现跨用户数据交叉或泄露
- **SC-006**: 全部用户故事验收场景通过率 100%

## Assumptions

- 本系统部署在单一家庭中，家庭成员数量通常不超过 10 人，无需考虑大规模并发
- `member_type` 字段（member/guest）是本特性的核心权限区分：成员（member）拥有全部管理能力（创建成员、切换用户），访客（guest）仅能使用聊天功能
- 现有 SysUser 的 `type` 字段（admin/user）保持不变，不用于本特性的权限判断。~~历史参考：原计划按 type 字段映射 admin→member、user→guest，已取消~~ — 开发完成后全量清库，无需存量数据迁移
- 开发完成后全量清库：清空 PostgreSQL 所有业务表、MinIO、向量库、Langfuse 监控数据，初始化一个带声纹的管理员账户（anlin），由其创建其他测试账户（zhenghui 等），用于记忆隔离和过期时间的完整验收测试。无需存量数据迁移
- 所有用户一旦创建不可删除，该类型问题不予考虑
- 系统中不存在无声纹的用户：初始管理员自带声纹，新用户创建为原子操作（声纹注册与用户入库在同一请求中完成）
- 声纹注册 Gateway API 已存在且功能完整；用户创建时由后端 MemberService 内部调用 Gateway 声纹注册 API（不通过前端 voice API 端点），前端仅需将录音音频作为 `POST /api/v1/members/` 请求的一部分提交
- 现有 user_id 隔离机制覆盖所有数据模型（Message、UserMemory、LangGraph thread 等），无需额外改造
- 访客有效期固定 7 天（从创建时间起算），不可自定义
- 头像使用用户名首字符大写 + 颜色方案，不支持自定义头像上传；用户名即显示名称，无额外 display_name 字段

## Clarifications

### Session 2026-03-11

- Q: 现有用户的 member_type 迁移策略？ → A: 无需迁移。开发完成后全量清库，初始化一个带声纹的管理员账户（anlin），由其创建其他测试账户
- Q: 切换后令牌的权限范围？ → A: 切换用户不改变登录身份，不颁发新令牌。切换仅影响 Message 等数据表的读写目标 user_id，当前登录用户（admin/member）始终保持完整权限。本质是"选择当前操作的目标用户"而非"模拟登录"
- Q: 过期访客的数据保留策略？ → A: 永久保留。过期仅禁止登录，消息和记忆数据不清理。成员不可切换到过期访客视角（前端列表灰色展示、不可点击），如需查看过期访客历史数据可直接查询数据库
- Q: 成员管理面板的 UI 入口位置？ → A: 底部输入框左侧用户头像 + 切换按钮 → 弹出模态框。模态框包含：(1) 可切换用户列表（有效用户正常展示，过期访客灰色展示在底部仅供查看状态，不可点击切换）；(2) 添加用户分步引导：步骤1 选择类型（成员/访客）+ 填写用户名密码，步骤2 注册声纹，声纹注册成功后完成创建。点击列表中的用户即可切换
- Q: 创建用户时声纹步骤是否可跳过？ → A: 不可跳过。前端向导步骤2必须完成录音，提交后后端原子完成声纹注册+用户入库，失败可重试但不能跳过
- Q: 切换视角后目标 user_id 在前端如何持久化？ → A: localStorage + Zustand store。刷新页面保持当前视角，新标签页共享目标用户。不使用 URL 参数（避免暴露 user_id）或后端 session（避免增加后端复杂度）
- Q: 创建用户时 UI 类型选项标签？ → A: 显示"成员/访客"，与 member_type 字段值 member/guest 完全对齐。不使用"admin/guest"避免与 legacy type 字段混淆
- Q: 成员"编辑"或"删除"用户？ → A: 不支持编辑和删除。成员只能创建用户，用户一旦创建不可删除。该类型问题不予考虑
- Q: 声纹录音最低有效时长？ → A: 最低 10 秒。前端在录音不足 10 秒时禁用停止/提交按钮，进度条标注 10 秒最低线
- Q: display_name 是否必填？ → A: 取消 display_name 字段。用户名（username）即显示名称，头像首字母取自用户名首字符大写（如 dantsinghua → D）。不引入额外展示字段，极简数据模型
