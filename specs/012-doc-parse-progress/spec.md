# Feature Specification: 文档解析进度展示与状态透传

**Feature Branch**: `012-doc-parse-progress`
**Created**: 2026-03-05
**Status**: Draft
**Input**: 用户上传 PDF 文档后，AI 解析过程中（3-5 分钟）页面无任何进度反馈；Gateway 返回的 5 种状态仅处理了 2 种；frpc 隧道偶发抖动导致轮询报错但实际任务已完成

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 解析中实时查看页级进度 (Priority: P1)

用户上传一份多页 PDF（如 36 页论文），AI 开始通过文档解析服务逐页处理。用户在聊天区域底部看到实时进度条，显示当前正在解析第几页、共多少页、百分比，而不是只看到闪烁的光标。

**Why this priority**: 解析 3-5 分钟期间用户完全无感知是最核心的体验问题。进度可见性直接决定用户是否会误以为系统卡死而刷新页面。

**Independent Test**: 上传一份多页 PDF，观察聊天区域底部出现进度条，进度数字逐步递增至完成后自动消失，全程无需其他功能依赖。

**Acceptance Scenarios**:

1. **Given** 用户上传了一份 36 页 PDF 并发送消息, **When** AI 开始解析, **Then** 聊天区域底部出现进度提示，显示文件名和"等待解析"状态
2. **Given** 解析正在进行中, **When** 每解析完一页, **Then** 进度条更新当前页数和百分比（如"第 5/36 页 14%"）
3. **Given** 解析全部完成, **When** 所有页面解析成功, **Then** 进度条变为"解析完成"提示，短暂停留后自动消失，AI 输出解析结果

---

### User Story 2 — 部分完成时展示已有结果 (Priority: P2)

文档解析过程中，解析引擎因资源问题崩溃，已成功解析了前 8 页但后续页面失败。系统应展示已解析的部分内容，而不是丢弃所有结果或显示完全失败。

**Why this priority**: 长文档解析中引擎故障是真实场景，丢弃已有结果会浪费用户等待时间，降低系统可用性感受。

**Independent Test**: 在解析引擎不稳定的环境下上传文档，验证系统返回已成功解析的页面内容并展示部分完成提示。

**Acceptance Scenarios**:

1. **Given** 解析进行中且前 8 页已成功, **When** 解析引擎崩溃导致后续页面失败, **Then** 进度条显示橙色"部分完成"警告，附解析成功的页数信息
2. **Given** 部分完成状态, **When** AI 获取解析结果, **Then** AI 输出已成功解析的 8 页内容，并附说明"部分页面解析失败"
3. **Given** 部分完成状态, **When** 解析服务返回建议信息, **Then** 系统将建议信息透传给用户（如"已解析页面可通过 result 接口获取"）

---

### User Story 3 — 网络抖动不中断解析流程 (Priority: P3)

文档解析过程中，内网穿透隧道发生短暂断连（1-3 秒），系统自动重试网络请求，用户无感知，解析流程不中断。

**Why this priority**: 网络抖动是已知的基础设施问题，如果每次抖动都导致解析报错，用户需要重新上传文档重新等待，体验极差。

**Independent Test**: 在解析过程中模拟网络中断后恢复，验证系统自动重试且解析结果正常返回。

**Acceptance Scenarios**:

1. **Given** 解析正在轮询进度中, **When** 网络隧道短暂断连后恢复, **Then** 系统自动重试轮询请求，前端进度条不出现错误提示
2. **Given** 网络隧道持续不可用超过重试次数, **When** 所有重试均失败, **Then** 系统显示明确的网络错误信息

---

### Edge Cases

- 用户同时上传多份文档时，进度条应展示当前正在解析的文档进度
- 解析进度事件到达前端时，如果用户已切换到其他页面再切回，进度条应恢复当前状态
- 文档解析完全失败（文件格式不支持、服务不可用等）时，进度条显示错误后在合理时间内自动消失
- 解析轮询超时（超过最大等待时间）时，应视为失败并通知用户

## Requirements *(mandatory)*

### Functional Requirements

**进度推送**:
- **FR-001**: 系统 MUST 在文档解析轮询过程中，每次获取到状态更新后实时推送进度事件给对应用户
- **FR-002**: 进度事件 MUST 包含：任务标识、当前状态、页级进度（当前页/总页数）、文件名
- **FR-003**: 系统 MUST 在解析完成、部分完成、失败、超时时各推送一次终态事件

**状态透传**:
- **FR-004**: 系统 MUST 准确透传解析服务返回的全部 5 种状态：等待中、解析中、全部完成、部分完成、失败
- **FR-005**: 系统 MUST 透传解析服务返回的建议信息字段（部分完成和失败时有值）
- **FR-006**: 部分完成状态下，系统 MUST 获取并缓存已成功解析的页面内容，不丢弃已有结果
- **FR-007**: 系统 MUST NOT 在客户端侧增加解析引擎恢复、心跳监控或 VRAM 检测等逻辑（由解析服务内部处理）

**网络容错**:
- **FR-008**: 状态轮询遇到网络连接错误时，系统 MUST 自动重试（最多 3 次，每次间隔 2 秒）
- **FR-009**: 网络重试 MUST 仅针对连接级错误（如连接断开、服务器无响应），不对超时或业务错误重试

**前端展示**:
- **FR-010**: 解析进度提示 MUST 展示在聊天消息列表底部，与现有系统提示（视频分析提示、上下文压缩提示）位置一致
- **FR-011**: 不同状态 MUST 有视觉差异化展示（等待中/解析中/完成/部分完成/失败分别使用不同图标和颜色）
- **FR-012**: 终态提示 MUST 在短暂停留后自动消失（完成状态 1.5 秒，其他终态 3 秒）

### Key Entities

- **解析进度事件**: 解析服务每次状态更新时产生的实时事件，包含任务标识、状态、页级进度、文件名、建议信息、错误信息
- **文档解析任务状态**: 由解析服务定义的 5 种状态（等待中/解析中/全部完成/部分完成/失败），系统透明传递

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户上传多页文档后，3 秒内能看到解析进度提示出现在聊天区域
- **SC-002**: 解析过程中用户可以实时看到页级进度更新（精确到页），进度数字与实际解析进展一致
- **SC-003**: 网络隧道短暂断连（3 秒以内）不会导致解析流程中断或向用户展示错误
- **SC-004**: 部分完成场景下，用户仍能获得已成功解析的页面内容，不会看到"完全失败"的错误提示
- **SC-005**: 全部 5 种解析状态在前端均有明确的视觉反馈，用户能区分正在进行、已完成和异常情况

## Assumptions

- 解析服务（LLM Gateway）已实现完整的 5 种状态机和引擎崩溃自动恢复机制，客户端侧不需要重复实现
- 现有的 SSE 事件推送管道（Redis Pub/Sub → EventSource）稳定可用，无需新增通信架构
- 前端全局状态管理和 SSE 事件分发机制已就绪
- 网络抖动为偶发且短暂的（通常 1-3 秒），3 次重试足以覆盖绝大多数场景

## Dependencies

- 解析服务（LLM Gateway）提供标准的 5 种状态响应和页级进度信息
- 现有 SSE 事件推送管道正常工作
- 现有聊天消息列表组件的内联提示模式可复用

## Scope Boundaries

**包含**:
- 文档解析过程中的实时进度推送和前端展示
- Gateway 5 种状态的完整透传
- frpc 网络隧道抖动的轮询重试容错

**不包含**:
- 解析引擎的崩溃恢复、心跳监控、VRAM 管理（由 Gateway 内部处理）
- 文档解析的核心功能改动（已在 011 特性中完成）
- 文档上传流程的改动
- 解析结果的 RAG 向量化流程改动
