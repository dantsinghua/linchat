# Feature Specification: 文档 SubAgent + 解析结果持久化 + 轻量 RAG

**Feature Branch**: `011-document-subagent-rag`
**Created**: 2026-03-05
**Status**: Draft
**Input**: 将文档解析从一次性使用升级为持久化存储 + 独立 SubAgent + 轻量 RAG 检索

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 文档解析结果缓存复用 (Priority: P1)

用户上传 PDF/DOCX 后，系统解析文档内容并持久化存储解析结果。同一文档再次提问时，系统直接使用已缓存的解析结果，无需重新耗时解析（从 2-5 分钟降至秒级）。即使原始文件因过期被清理，解析结果仍然保留可查。

**Why this priority**: 解决当前最大痛点——每次重复提问同一文档都需要重新解析 2-5 分钟。这是后续所有文档能力（搜索、比较、RAG）的基础。

**Independent Test**: 上传一个 PDF，首次解析后再次提问同一文档，验证秒级返回且内容一致。

**Acceptance Scenarios**:

1. **Given** 用户首次上传一个 PDF 文档, **When** 系统完成解析, **Then** 解析结果同时写入数据库和对象存储，两处内容一致
2. **Given** 同一文档已被解析过, **When** 用户再次对该文档提问, **Then** 系统在 1 秒内返回缓存的解析结果，不触发重新解析
3. **Given** 文档的原始文件因过期已被清理, **When** 用户查询该文档的解析内容, **Then** 系统仍能返回完整的解析结果
4. **Given** 解析结果写入对象存储成功但数据库写入失败, **When** 系统检测到不一致, **Then** 自动删除对象存储中的文件以保持一致性
5. **Given** 对象存储写入失败, **When** 系统检测到错误, **Then** 不写入数据库，向用户返回正常的解析结果（本次可用但不缓存）
6. **Given** 用户对已解析文档的解析质量不满意, **When** 用户要求重新解析, **Then** 系统清除旧的解析缓存和索引，重新执行完整解析流程并重建分块索引

---

### User Story 2 - 独立文档管理（列出 + 读取）(Priority: P1)

用户可以通过自然语言查询自己上传过的文档，包括列出文档列表、按文件名搜索、按时间筛选、查看文档完整解析内容。支持跨消息操作如"之前那篇文档再总结一下"。

**Why this priority**: 让用户能独立管理和回顾文档是核心交互能力，与 P1 缓存共同构成最小可用产品。

**Independent Test**: 上传若干文档后，通过自然语言询问"我上传过哪些文档"、"上次那篇论文的核心观点是什么"，验证正确返回。

**Acceptance Scenarios**:

1. **Given** 用户已上传多个文档, **When** 用户说"我上传过哪些文档", **Then** 系统列出所有文档，显示文件名、大小、上传时间、解析状态、原始文件可用性
2. **Given** 用户已上传多个文档, **When** 用户说"找一下量子计算的论文", **Then** 系统按文件名分词搜索，返回匹配的文档列表
3. **Given** 用户最近上传过一篇论文, **When** 用户说"之前那篇论文再总结一下", **Then** 系统找到最近的文档并读取其完整解析内容，生成摘要
4. **Given** 文档原始文件已过期, **When** 用户列出文档列表, **Then** 文档仍显示在列表中，标注"原始文件已过期"但"解析结果可用"
5. **Given** 用户指定时间范围, **When** 用户说"上周上传的文档有哪些", **Then** 系统按时间范围筛选返回匹配的文档

---

### User Story 3 - 文档内容搜索（轻量 RAG）(Priority: P2)

用户可以在所有已解析的文档中进行内容搜索。系统支持关键词搜索、语义搜索和混合搜索（关键词 + 语义 + 重排序），返回最相关的文档片段及其来源信息。

**Why this priority**: 语义搜索大幅提升文档查找精度，但依赖分块和 Embedding 基础设施，比文档管理更复杂。

**Independent Test**: 上传多篇文档后，搜索特定主题关键词，验证返回相关文档片段且来源标注正确。

**Acceptance Scenarios**:

1. **Given** 多篇文档已解析并完成索引, **When** 用户说"哪篇文档提到了金融监管", **Then** 系统使用混合搜索返回最相关的文档片段，包含文件名、匹配文本和相关度评分
2. **Given** 语义搜索服务暂时不可用, **When** 用户执行文档搜索, **Then** 系统自动降级为纯关键词搜索，仍然返回有用结果
3. **Given** 文档已解析但分块索引尚未完成, **When** 用户搜索文档内容, **Then** 系统使用已有的能力（关键词全文搜索）返回结果，不报错
4. **Given** 搜索无匹配结果, **When** 用户的查询与所有文档内容都不相关, **Then** 系统明确告知未找到匹配内容

---

### User Story 4 - 多文档综合操作 (Priority: P2)

用户可以要求系统对多篇文档进行综合操作，如比较两篇文档、汇总多篇文档的共同点等。

**Why this priority**: 跨文档操作是高级功能，依赖文档管理和读取能力完善后才能实现。

**Independent Test**: 上传两篇主题相关的文档，要求系统比较两篇文档的异同，验证输出包含两篇文档的内容分析。

**Acceptance Scenarios**:

1. **Given** 用户已上传两篇相关文档且均已解析, **When** 用户说"前两篇文档综合比较一下", **Then** 系统分别读取两篇文档内容，生成比较分析
2. **Given** 用户引用的文档中有一篇尚未解析, **When** 用户要求比较, **Then** 系统先触发解析，完成后再执行比较

---

### User Story 5 - 文档 SubAgent 从多模态 SubAgent 独立 (Priority: P1)

文档相关操作从多模态 SubAgent 中拆分出来，形成独立的文档 SubAgent。多模态 SubAgent 专注于图片/视频/音频分析，文档 SubAgent 专注于文档管理和检索。AI 能根据用户意图自动选择正确的 SubAgent 路由。

**Why this priority**: 架构拆分是所有文档功能的前提。拆分后多模态 SubAgent 更轻量，文档 SubAgent 可独立迭代。

**Independent Test**: 分别发送图片分析请求和文档查询请求，验证路由到正确的 SubAgent。

**Acceptance Scenarios**:

1. **Given** 用户上传了一张图片, **When** 用户说"看看这张图片", **Then** 请求路由到多模态 SubAgent（不经过文档 SubAgent）
2. **Given** 用户上传了一个 PDF, **When** 用户说"解析这个 PDF", **Then** 请求路由到文档 SubAgent
3. **Given** 用户没有上传新文件, **When** 用户说"之前上传的论文有哪些", **Then** 请求路由到文档 SubAgent
4. **Given** 用户说"搜索文档中关于量子计算的内容", **When** AI 处理请求, **Then** 请求路由到文档 SubAgent 使用搜索功能

---

### Edge Cases

- 用户在解析过程中对同一文档发起第二次解析请求时，系统采用幂等设计（相同附件 ID 产生相同存储路径），避免重复解析
- 解析结果为空（文档是纯图片扫描件无文字）时，系统提示用户文档未识别到文本内容
- 文档极大（超过 100 页），解析结果文本量巨大时，分块策略自动切分，返回结果按需截断并标注
- Embedding 生成持续失败（如模型服务不可用）时，不影响关键词搜索的可用性
- 部分解析（指定页码范围）时不使用缓存，因为缓存的是全文解析结果
- 数据库中解析内容为空但对象存储中有备份时，系统自动降级从对象存储读取
- 用户要求强制重新解析已缓存的文档时，系统清除旧缓存（数据库 + 对象存储 + 分块索引），重新执行完整解析流程并重建索引
- 系统不支持用户主动删除文档，文档生命周期仅由系统自动过期机制管理

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在文档解析完成后，将解析结果全文同时写入数据库和对象存储（双写），两处内容保持一致
- **FR-002**: 系统必须在检测到双写不一致时执行补偿操作（对象存储成功但数据库失败时删除对象存储文件）
- **FR-003**: 系统必须在同一文档再次被查询时优先使用数据库中的缓存结果，缓存命中时响应时间小于 1 秒
- **FR-004**: 系统必须在原始文件过期清理后仍保留解析结果和搜索索引，不影响查询
- **FR-005**: 系统必须支持按文件名分词搜索（空格分隔的多关键词 AND 匹配）
- **FR-006**: 系统必须支持按上传时间范围筛选文档列表
- **FR-007**: 系统必须支持多种排序方式（最新、最早、文件名、文件大小）
- **FR-008**: 系统必须在解析完成后自动将文档内容分块并生成语义向量索引
- **FR-009**: 系统必须支持三种文档内容搜索模式：关键词搜索、语义向量搜索、混合搜索（关键词 + 语义 + 重排序）
- **FR-010**: 系统必须在语义搜索不可用时自动降级为纯关键词搜索
- **FR-011**: 系统必须将文档管理功能作为独立 SubAgent 运行，从多模态 SubAgent 中分离
- **FR-012**: 多模态 SubAgent 必须在文档功能分离后仅保留图片/视频/音频分析能力
- **FR-013**: 系统必须支持读取指定文档的完整解析内容（通过文档标识符）
- **FR-014**: 系统必须在索引生成失败时标记状态并定时自动重试
- **FR-015**: 系统必须确保所有文档查询按用户维度隔离，用户只能访问自己的文档
- **FR-016**: 系统必须在文档搜索结果中包含来源信息（文件名、匹配文本、相关度评分）
- **FR-017**: 系统必须支持对已解析文档进行强制重新解析，清除旧缓存（解析结果 + 对象存储备份 + 分块索引）后重新执行完整解析并重建索引
- **FR-018**: 系统不支持用户主动永久删除文档，文档生命周期仅由系统自动过期机制管理

### Key Entities

- **文档附件 (MediaAttachment)**: 现有实体扩展，新增解析结果全文、解析时间、解析结果大小、对象存储备份路径、索引生成状态。代表用户上传的文档及其解析后的内容
- **文档内容分块 (DocumentChunkEmbedding)**: 新增实体，与文档附件为多对一关系。存储文档的分块文本和对应的语义向量，用于支持语义搜索。包含分块序号、分块文本、语义向量、所属用户标识（冗余字段用于加速查询）
- **文档 SubAgent**: 独立的 AI 子代理，负责文档列表查询、文档内容读取、文档内容搜索、新文档解析四项能力

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 已解析文档的重复查询响应时间从 2-5 分钟降至 1 秒以内（缓存命中场景）
- **SC-002**: 用户能够通过自然语言成功找到已上传的文档，文档列表查询成功率 100%
- **SC-003**: 混合搜索（关键词 + 语义）在相关文档存在时的召回率不低于 80%
- **SC-004**: 原始文件过期后，95% 以上的解析结果仍可正常读取和搜索
- **SC-005**: 文档 SubAgent 与多模态 SubAgent 路由准确率不低于 95%（文档请求不误路由到多模态，反之亦然）
- **SC-006**: 索引生成失败后自动重试成功率不低于 90%（排除外部服务持续不可用的情况）
- **SC-007**: 全量测试通过，无回归

## Clarifications

### Session 2026-03-05

- Q: 用户是否需要能主动永久删除文档及其所有解析数据？ → A: 不支持。页面没有此操作，仅依赖系统自动过期机制。
- Q: 用户是否可以对已解析的文档强制重新解析（刷新缓存）？ → A: 支持。document_parse 增加强制刷新选项，清除旧缓存并重建索引。

## Assumptions

- 复用现有的 Embedding 模型和维度（1024 维），与记忆模块共享同一 Embedding 服务
- 复用现有的 GPU 互斥锁机制，文档解析和 Embedding 生成遵循同样的资源调度
- 文档分块策略采用 Markdown 标题分段 + 段落拆分 + 长段切分的层级方案
- 混合搜索的默认权重为语义 70% + 关键词 30%，可通过配置调整
- 分块参数默认 800 字符/块、100 字符重叠，可通过配置调整
- 文档 SubAgent 的超时时间沿用文档解析场景的长超时设置
- 中文分词搜索复用现有的 jieba 分词配置
