# 研究文档：模型配置管理

**特性**：003-model-config
**日期**：2026-01-29

## 研究项目

### R1：SM4 加密集成方案

**决策**：复用现有 `apps/users/crypto.py` 中的 `sm4_encrypt` / `sm4_decrypt` 函数

**理由**：
- 项目已有完整的 SM4 加密实现（ECB 模式 + PKCS7 填充 + Base64 编码）
- 密钥来源统一（`SM4_SECRET_KEY` 环境变量）
- `sm4_encrypt_safe` / `sm4_decrypt_safe` 提供异常安全的变体

**备选方案**：
- 新建独立加密模块：不必要，现有实现完全满足需求
- 使用 Django 内置加密：不符合国密 SM4 要求

### R2：数据库迁移策略 — 环境变量数据种子

**决策**：Django migration 中通过 `RunPython` 预置 2 条记录，初始值从环境变量读取（仅迁移时读取一次）

**理由**：
- 迁移时自动从 `LLM_API_BASE`、`LLM_API_KEY`、`LLM_MODEL_NAME` 读取现有配置
- API Key 写入数据库前先经过 SM4 加密
- 迁移完成后环境变量不再使用（数据库为唯一来源）
- embedding 模型记录初始值留空（当前无环境变量配置）

**备选方案**：
- 手动 SQL 脚本：不易版本控制，不符合 Django 最佳实践
- manage.py 自定义命令：额外一步，容易遗忘

### R3：LLM 客户端配置热更新方案

**决策**：移除 `get_llm()` 的 `@lru_cache` 装饰器，每次调用从数据库读取最新配置

**理由**：
- 当前 `agent.py` 使用 `@lru_cache(maxsize=1)` 缓存 `ChatOpenAI` 实例
- 模型配置修改后需要立即生效，缓存会导致使用旧配置
- 模型配置读取是简单的主键查询（2 条记录），性能开销极小
- 可考虑短 TTL 缓存（如 30s），但鉴于记录数量极少，直接查询更简单可靠

**备选方案**：
- 保留 `lru_cache` + 手动失效：增加复杂度，缓存失效是已知的难题
- Redis 缓存模型配置：对 2 条记录过度设计
- 信号机制（post_save 清缓存）：增加耦合，不值得

### R4：前端 NULL vs 0 区分方案

**决策**：前端表单中，选填数值字段使用受控输入：空字符串 `""` 表示 NULL，数字 `0` 表示零值。提交时将 `""` 转为 `null`

**理由**：
- HTML input 的空值天然为 `""`，与 `"0"` 字符串可区分
- 提交前将 `""` 转为 JSON `null`，`"0"` 转为数字 `0`
- 后端序列化器中选填字段设置 `required=False, allow_null=True`
- 这是最直观的用户交互方式，无需额外 UI 元素

**备选方案**：
- 单独的"清空"按钮：过度设计，增加 UI 复杂度
- 三态开关（未设置/开启/值）：不直观

### R5：前端页面路由与导航入口

**决策**：新增 `/settings` 路由页面，在聊天页面侧边栏底部添加设置入口图标（仅管理员可见）

**理由**：
- 当前前端只有 `/login`、`/chat`、`/401` 三个路由
- 设置功能独立于聊天，应有独立页面
- 侧边栏底部放置设置图标是常见的 SaaS 应用模式
- Next.js App Router 下新增 `app/settings/page.tsx` 即可
- 非管理员用户看不到入口，通过 URL 直接访问时统一路由到已有的错误页面

**备选方案**：
- 弹窗/抽屉：配置字段较多，弹窗空间不够
- 仅通过 URL 访问：不符合可发现性原则

### R6：API Key 脱敏展示策略

**决策**：后端返回时对 API Key 进行脱敏（显示前 4 位 + `****` + 后 4 位），前端不存储完整密钥

**理由**：
- 脱敏在后端序列化器中完成，前端永远拿不到完整密钥
- GET 请求返回脱敏值，PUT 请求中如果 API Key 字段为脱敏值（包含 `****`），则保留数据库中的原值不更新
- 仅当 PUT 请求中 API Key 是新值（不包含 `****`）时才更新

**备选方案**：
- 前端脱敏：完整密钥经过网络传输，有泄露风险
- 不返回 API Key：用户无法确认是否已配置

## 总结

所有研究项目均已解决，无待澄清项残留。可进入阶段 1 设计。
