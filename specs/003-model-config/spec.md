**创建日期**：2026-01-29
**状态**：草稿
**输入**：将模型配置从环境变量迁移到数据库，构建模型配置管理页面，支持查看和修改模型参数

### 2026-01-29

## 用户场景与测试 *(必填)*

### 用户故事 1 — 查看模型配置（优先级：P1）

管理员打开模型配置页面，查看系统当前配置的语言模型和 embedding 模型的详细参数，包括模型名称、API 地址、上下文窗口大小、采样参数等信息。API Key 以脱敏形式展示（仅显示前 4 位和后 4 位）。页面按模型类型分为两张配置卡片（language 卡片和 embedding 卡片），每张卡片展示对应模型的完整参数。

**优先级原因**：查看是所有管理操作的基础，管理员需要先了解当前配置状态才能进行修改。即使没有修改功能，查看功能本身也提供了配置透明性的价值。

**独立测试**：可以通过打开配置页面，验证所有模型记录及其参数是否正确展示来独立测试。

**验收场景**：

1. **假设** 系统已预置语言模型和 embedding 模型记录，**当** 管理员打开模型配置页面，**则** 页面按模型类型展示两张配置卡片（language 卡片和 embedding 卡片），每张卡片包含名称、URL、上下文窗口等参数
2. **假设** 模型记录包含 API Key，**当** 页面展示模型信息，**则** API Key 以脱敏形式显示（如 `sk-t****cxyz`），不暴露完整密钥
3. **假设** 模型的选填参数（如 `temperature`）为空，**当** 页面展示该模型信息，**则** 选填参数显示为空或"未设置"状态，而非显示 0

---

### 用户故事 2 — 修改模型配置（优先级：P1）

管理员在配置卡片上点击编辑，修改模型参数（如更换 API 地址、更新 API Key、调整上下文窗口大小或采样参数），保存后配置立即生效，无需重启服务。

**优先级原因**：修改功能是本特性的核心价值——将模型配置从需要修改环境变量并重启的方式，改为在线即时修改。

**独立测试**：可以通过修改某个模型参数并保存，然后刷新页面验证参数已更新来独立测试。

**验收场景**：

1. **假设** 管理员在配置页面查看某个模型，**当** 管理员修改模型名称和 URL 并保存，**则** 系统校验必填字段后保存成功，页面展示更新后的配置
2. **假设** 管理员修改模型配置，**当** 必填字段（`name`、`url`、`api_key`、`max_context_window`、`max_input_tokens`、`max_output_tokens`）为空，**则** 系统提示校验错误，阻止保存
3. **假设** 管理员修改选填参数 `temperature`，**当** 管理员将值设为 0，**则** 系统保存值为 0（而非 NULL），后续 API 调用时传入 `temperature=0`（详见 FR-006 存储语义、FR-007 调用行为）
4. **假设** 管理员修改选填参数 `temperature`，**当** 管理员清空该字段，**则** 系统保存值为 NULL，后续 API 调用时不传入该参数（详见 FR-006 存储语义、FR-007 调用行为）
5. **假设** 管理员保存了模型配置修改，**当** 下一次 AI 对话请求发生，**则** 系统使用最新保存的模型配置，无需重启服务

---

### 用户故事 3 — 选填参数的精确控制（优先级：P2）

管理员需要精确控制模型的采样参数，区分"不设置"（使用模型默认值）和"设置为 0"两种不同语义，确保 API 调用行为符合预期。

**优先级原因**：选填参数的 NULL 与 0 区分是保证模型调用行为正确性的关键，但依赖于基础的查看和修改功能。

**独立测试**：可以通过分别设置 `temperature` 为 0 和清空 `temperature`，然后检查 API 请求参数来验证。

**验收场景**：

1. **假设** 模型的 `temperature` 为 NULL，**当** 系统构造 API 请求，**则** 请求参数中不包含 `temperature` 字段（详见 FR-007）
2. **假设** 模型的 `temperature` 为 0，**当** 系统构造 API 请求，**则** 请求参数中包含 `temperature: 0`（详见 FR-007）
3. **假设** 管理员在表单中操作选填参数，**当** 管理员需要区分"清空"和"设为 0"，**则** 表单提供明确的交互方式让管理员可以分别执行这两种操作（详见 FR-006）

---

### 用户故事 4 — 现有聊天功能对接新配置（优先级：P1）

系统现有的 AI 聊天功能从数据库读取模型配置替代环境变量，确保聊天功能正常运行，用户无感知切换。

**优先级原因**：这是迁移的核心目标——现有功能必须无缝切换到数据库配置，否则整个特性没有意义。

**独立测试**：可以通过在数据库中配置正确的模型信息，然后发起一次 AI 对话验证是否正常响应来测试。

**验收场景**：

1. **假设** 模型配置已从环境变量迁移到数据库，**当** 用户发起 AI 对话，**则** 系统从数据库获取激活的语言模型配置，正常完成对话
2. **假设** 数据库中存在多个模型记录，**当** 系统需要调用语言模型，**则** 系统使用 `is_active=true` 且 `type=language` 的模型配置
3. **假设** 管理员在配置页面更新了模型 API Key，**当** 用户随后发起对话，**则** 系统使用更新后的 API Key 进行调用

---

### 边界情况

- ~~当所有模型的 `is_active` 都为 false 时~~ 不适用：M1a 阶段两条记录始终激活，`is_active` 由系统管理不可编辑
- 当管理员输入无效的 API URL 或 API Key 时，系统是否在保存时验证连通性？**处理方式**：保存时仅做基础校验（非空 + 最大长度 500 字符），不验证 URL 格式合法性（允许任意非空字符串），不做连通性验证（避免保存延迟），连通性问题在实际调用时暴露。API Key 必须至少 12 个字符。`name` 字段仅做非空 + 最大 100 字符长度校验，无格式限制
- 当管理员输入 `max_context_window` 等整数字段时是否允许小数？**处理方式**：不允许，`max_context_window`、`max_input_tokens`、`max_output_tokens`、`embedding_dimensions` 必须为非零正整数，后端拒绝小数输入，前端表单使用 `type="number" step="1"` 限制
- 当数值参数超出合理范围时如何处理？**处理方式**：后端校验参数范围（如 `temperature` 0~2，`top_p` 0~1），超出范围返回校验错误
- 当管理员同时在两个浏览器窗口修改同一模型配置时如何处理？**处理方式**：后保存的覆盖先保存的（最后写入优先），不做乐观锁

## 需求 *(必填)*

### 功能需求

> **编号说明：** FR-016/FR-017 为后续补充的权限需求，编号追加在末尾。FR-014/FR-015 为原有需求。编号不连续不影响追溯。

- **FR-001**：系统必须将模型配置信息持久化存储在数据库中，完全移除环境变量支持，数据库为唯一配置来源
- **FR-002**：系统必须预置两条模型记录：一条 `language` 类型，一条 `embedding` 类型
- **FR-003**：系统必须提供独立的模型配置设置页面，在用户名与退出按钮之间新增"模型配置"按钮（仅管理员可见），点击跳转到模型配置页面，按模型类型以配置卡片形式展示所有模型记录（language 卡片和 embedding 卡片）
- **FR-004**：系统必须提供模型配置修改功能，管理员可编辑模型的所有参数（`is_active` 和 `type` 除外，由系统管理）
- **FR-005**：系统必须对必填字段进行校验：`name`、`url`、`api_key`、`max_context_window`、`max_input_tokens`、`max_output_tokens` 不可为空。注意：`type` 在初始迁移时为必填，但 PUT 接口中为只读字段（不可修改），故 PUT 校验不包含 `type`。**补充**：PUT 时 `api_key` 允许传入脱敏值（含 `****`），由 Service 层判定后保留原值，脱敏值视为有效值通过必填校验。**判断规则**：Service 层使用 `"****" in api_key` 判定脱敏值。已知极端情况：新密钥恰好包含 `****` 子串时会被误判为保留原值——此概率极低，可接受
- **FR-006**：系统必须区分选填参数的 NULL（不设置）和 0（设置为零）两种存储状态。（本条定义**存储语义**，调用行为详见 → FR-007）
- **FR-007**：系统必须在构造 API 请求时，仅传入非 NULL 的选填参数。（本条定义**调用行为**，存储语义详见 → FR-006）
- **FR-008**：系统必须对 API Key 进行加密存储（SM4 加密）
- **FR-009**：系统必须在前端展示时对 API Key 进行脱敏处理（前 4 位 + `****` + 后 4 位）。当 API Key 长度 ≤ 8 时，全部脱敏为 `****`
- **FR-010**：系统必须在配置修改保存后，下一次请求即使用新配置（无缓存设计，每次请求直接从数据库读取最新配置），无需重启服务
- **FR-011**：系统必须将现有聊天功能的模型配置读取从环境变量切换到数据库
- **FR-012**：系统必须对参数进行范围校验：`api_key` 为字符串输入，最少 12 个字符；`max_context_window`、`max_input_tokens`、`max_output_tokens` 必须为非零正整数（> 0，不接受小数）；`temperature`: [0, 2] 闭区间；`top_p`: [0, 1] 闭区间；`frequency_penalty`: [-2, 2] 闭区间；`presence_penalty`: [-2, 2] 闭区间——以上浮点参数为选填，非空时必须满足对应区间，也可置空（NULL）；`embedding_dimensions`: 非零正整数（> 0）或 NULL
- **FR-013**：系统禁止提供模型记录的增加和删除功能——API 仅支持 GET 和 PUT 方法，其他所有方法（POST、DELETE、PATCH 等）返回 405
- **FR-016**：系统必须限制模型配置 API（GET/PUT `/api/v1/models/`）仅允许管理员用户访问，非管理员用户请求返回 403
- **FR-017**：系统必须限制模型配置前端页面仅对管理员用户可见——在用户名与退出按钮之间仅对管理员显示"模型配置"按钮，非管理员用户看不到该入口；非管理员通过 URL 直接访问 `/settings` 页面时，统一路由到已有的错误页面
- **FR-014**：系统必须提供获取当前激活模型的能力，按 `type` 和 `is_active` 筛选。此能力通过服务层内部调用实现（`get_active_model(type)`），不通过 API 端点暴露，不受 FR-016 管理员权限限制。`is_active` 由系统管理，管理员不可编辑；两条预置记录始终为激活状态
- **FR-015**：系统必须计算有效上下文窗口（`int(max_context_window * 0.9)`，向下取整），作为模型计算属性暴露。**注意：此属性供 M1b 上下文管理特性使用，M1a 阶段的模型 API 调用直接使用 `max_context_window` 原始值**

### 错误码约定

> 统一响应格式 `{code, message, data}` 中 `code` 字段值定义如下：

| code | 含义 | 说明 |
|------|------|------|
| 0 | 成功 | 遵循宪法 1.2 统一响应格式 |
| 400 | 参数校验失败 | 必填字段缺失、范围超限等 |
| 401 | 未认证 | 未登录或登录已过期 |
| 403 | 权限不足 | 非管理员用户访问 |
| 404 | 资源不存在 | 模型 ID 无效 |
| 405 | 方法不允许 | POST/DELETE/PATCH 等禁止方法 |
| 500 | 服务器内部错误 | 未预期的异常 |

### 关键实体

- **模型配置（Model）**：代表一个 AI 模型的完整配置信息，包含连接信息（名称、URL、API Key）、容量参数（上下文窗口、输入/输出 token 限制）、采样参数（`temperature`、`top_p`、`frequency_penalty`、`presence_penalty`）。按类型分为 `language`（语言模型）和 `embedding`（向量模型）。每个模型有激活状态标识，系统通过类型和激活状态定位当前使用的模型。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：管理员可以在配置页面内完成模型参数的查看和修改，全程无需接触服务器环境变量或重启服务
- **SC-002**：模型配置修改保存后，下一次 AI 对话请求即使用新配置，生效延迟不超过 1 次请求
- **SC-003**：API Key 在前端任何展示场景中均为脱敏状态，完整密钥仅存在于加密存储中
- **SC-004**：选填参数的 NULL 和 0 在存储、展示、API 调用三个环节均能正确区分，行为一致（详见 FR-006 存储语义、FR-007 调用行为）
- **SC-005**：现有聊天功能切换到数据库配置后，用户对话体验无变化，功能正常

## 假设

- 系统当前仅需要管理少量模型记录（language 和 embedding 各一条），不需要批量管理能力
- 所有接入的模型 API 遵循 OpenAI API 兼容标准
- 系统已有用户角色体系，User 表包含 `type` 字段（值为 `admin` 或 `user`），模型配置页面和 API 仅允许 `type=admin` 的用户访问。权限类基于 `User.type == 'admin'` 判定，不使用 Django 内置 `is_staff`/`is_superuser`
- 数据迁移时，现有环境变量中的模型配置将作为初始数据写入数据库
- `embedding_dimensions` 字段仅对 embedding 类型模型有意义，language 类型模型该字段始终为 NULL

## 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 模型配置填写错误导致 API 调用失败 | 聊天功能不可用 | 前端必填校验 + 后端参数范围校验 |
| API Key 泄露 | 安全风险，密钥被滥用 | SM4 加密存储 + 前端脱敏展示 |
| 环境变量迁移遗漏 | 部分配置丢失 | 迁移脚本 + 迁移前后对比验证 |
